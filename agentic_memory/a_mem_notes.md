# A-Mem: Agentic Memory for LLM Agents

**Authors:** Wujiang Xu, Zujie Liang, Kai Mei, Hang Gao, Juntao Tan, Yongfeng Zhang (Rutgers University, AIOS Foundation)

**Paper:** arXiv:2502.12110 (NeurIPS submission) | https://github.com/WujiangXu/AgenticMemory | https://github.com/WujiangXu/A-mem-sys

---

## The Problem

Existing memory systems for LLM agents suffer from fundamental rigidity:

1. **Predefined structures** — systems like MemGPT, MemoryBank require developers to predefine memory storage structures, specify when to write, and establish retrieval timing. The agent's memory operations are hardcoded into the workflow.
2. **Graph databases (e.g., Mem0)** — attempt structured organization using graph DBs, but their **predefined schemas and relationships** limit adaptability. When an agent learns something novel, the system can only categorize within preset frameworks — it cannot forge new organizational patterns.
3. **Fixed operations** — existing systems cannot adapt their memory organization to new tasks or domains, severely restricting generalization in long-term interactions.

---

## The Core Idea: Zettelkasten for LLMs

A-Mem draws inspiration from the **Zettelkasten method** — a knowledge management system invented by sociologist Niklas Luhmann (who used it to produce 70+ books). Key principles:

- **Atomic notes**: Each piece of knowledge is a self-contained unit
- **Flexible linking**: Notes connect through meaningful relationships (not predefined categories)
- **Emergent organization**: Structure arises organically from connections between notes, not from top-down taxonomy
- **Boxes**: Related memories cluster into "boxes" through shared links — like Zettelkasten's index cards grouped in boxes, but a single memory can belong to multiple boxes simultaneously

The key innovation: **the memory system itself is agentic** — it autonomously decides how to organize, link, and evolve memories, rather than following fixed rules.

---

## Architecture: Three Core Operations

### Operation 1: Note Construction

When a new interaction occurs, the system constructs a **structured memory note** with 7 attributes:

```
m_i = {c_i, t_i, K_i, G_i, X_i, e_i, L_i}
```

| Attribute | Description | Source |
|---|---|---|
| `c_i` (content) | Raw interaction content | Direct from interaction |
| `t_i` (timestamp) | When the interaction happened | System clock |
| `K_i` (keywords) | Key concepts extracted | **LLM-generated** |
| `G_i` (tags) | Categorical labels for classification | **LLM-generated** |
| `X_i` (contextual description) | Rich semantic summary | **LLM-generated** |
| `e_i` (embedding) | Dense vector for similarity matching | Text encoder (all-MiniLM-L6-v2) |
| `L_i` (links) | Set of connected memories | Generated by link generation step |

**Process:**
1. LLM generates keywords, tags, and contextual description in a single call using structured JSON prompt:
   ```
   K_i, G_i, X_i ← LLM(c_i || t_i || P_s1)
   ```
2. Embedding computed over concatenation of all textual components:
   ```
   e_i = f_enc[concat(c_i, K_i, G_i, X_i)]
   ```

The multi-faceted note structure captures different aspects of the memory, facilitating both nuanced organization and computationally efficient retrieval.

---

### Operation 2: Link Generation

When a new note `m_n` is added, the system automatically discovers and establishes connections:

1. **Embedding retrieval**: Compute cosine similarity between `m_n`'s embedding and all existing memories:
   ```
   s_{n,j} = (e_n · e_j) / (|e_n| |e_j|)
   ```

2. **Top-k candidates**: Select the k most similar memories:
   ```
   M_near = {m_j | rank(s_{n,j}) ≤ k, m_j ∈ M}
   ```

3. **LLM analysis**: Prompt the LLM to analyze whether meaningful connections exist based on shared attributes, causal relationships, and conceptual connections:
   ```
   L_i ← LLM(m_n || M_near || P_s2)
   ```

**Why two steps?** Embedding similarity is a fast initial filter for scalability, but the LLM can identify **subtle patterns, causal relationships, and conceptual connections** that go beyond cosine similarity. This mirrors the Zettelkasten principle of flexible linking — connections emerge organically from content.

---

### Operation 3: Memory Evolution

The most novel component. After creating links for a new memory, the system **updates existing memories** based on the new information:

```
m*_j ← LLM(m_n || M_near\m_j || m_j || P_s3)
```

For each memory `m_j` in the nearest neighbor set, the LLM decides whether to:
- **Strengthen connections**: Add new links, update tags of the new memory
- **Update neighbors**: Revise the context and tags of existing memories based on new understanding

The evolved memory **replaces** the original in the memory store. Over time, this creates increasingly sophisticated knowledge structures — memories develop **higher-order patterns and attributes** that weren't present in any individual interaction.

This mimics how human learning works: when you learn something new, it doesn't just sit in isolation — it reshapes your understanding of things you already knew.

---

### Memory Retrieval

Given a query `q`:
1. Compute query embedding: `e_q = f_enc(q)`
2. Compute cosine similarity against all memory embeddings
3. Return top-k most relevant memories
4. **Linked retrieval**: When a memory is retrieved, linked memories from the same "box" are also automatically accessed — providing connected context without additional computation

- **Multi-Hop reasoning**: At least 2x better than baselines, demonstrating the value of interconnected memory networks for complex reasoning chains.

## Ablation Study (GPT-4o-mini)

| Configuration | Multi-Hop F1 | Temporal F1 | Open Domain F1 | Single Hop F1 | Adversarial F1 |
|---|---|---|---|---|---|
| w/o Link Gen & Memory Evolution | 9.65 | 24.55 | 7.77 | 13.28 | 15.32 |
| w/o Memory Evolution only | 21.35 | 31.24 | 10.13 | 39.17 | 44.16 |
| **Full A-Mem** | **27.02** | **45.85** | **12.14** | **44.65** | **50.03** |

**Both components matter, but differently:**
- **Link Generation is foundational** — removing both LG and ME causes the most dramatic drop (Multi-Hop: 9.65 vs 27.02)
- **Memory Evolution adds significant refinement** — Temporal F1 jumps from 31.24 to 45.85 with ME enabled
- The two modules are **complementary**: LG creates the structural foundation, ME refines and enriches it

## Hyperparameter Analysis (top-k retrieval)

Performance across k = 10, 20, 30, 40, 50 (GPT-4o-mini):

- **Multi-Hop**: Improves sharply from k=10 (19.91) to k=30 (26.97), plateaus at k=40 (27.02)
- **Temporal**: Steady improvement, peaks at k=40 (45.85)
- **Single Hop**: Continuously improves up to k=50 (44.55) — benefits from more context
- **Adversarial**: Peaks at k=40 (50.03), slight decline at k=50

**Trade-off**: Larger k provides richer context but may introduce noise and challenge the model's sequence processing capacity. Moderate k values (30-40) are optimal for most categories.

**Model-specific k settings:**
- GPT models: k=40-50 (can handle more context)
- Small models (Qwen2.5, Llama 3.2): k=10 (limited context processing capacity)

## Key Takeaways

1. **Agentic memory > static memory.** Letting the LLM autonomously organize, link, and evolve memories outperforms all predefined structures.
2. **Memory evolution is powerful.** Updating existing memories when new ones arrive — like how human learning reshapes prior understanding — creates higher-order knowledge structures.
3. **Link generation is foundational.** The two-step process (embedding retrieval → LLM analysis) catches both obvious and subtle connections between memories.
4. **Dramatic efficiency gains.** 85-93% fewer tokens than LOCOMO/MemGPT while outperforming both — structured atomic notes with rich context are more information-dense than raw dialog dumps.
5. **Small models benefit most.** The structured memory organization compensates for weaker reasoning in smaller models — rank 1.0 across all non-GPT models.
6. **Temporal reasoning sees the biggest improvement** — contextual descriptions, timestamps, and memory evolution help enormously with time-related questions (nearly 2x improvement over MemGPT).

---

## Limitations

1. Memory organization quality is influenced by the underlying LLM's capabilities — different models may generate varying contextual descriptions and connections.
2. Currently text-only — future work could extend to multimodal information (images, audio).
3. No error bars reported due to API cost constraints of running multiple trials.

## Cross-Paper Comparison

| Aspect | MemGPT | MemoryBank | LOCOMO | ReadAgent | MemoryOS | A-Mem |
|---|---|---|---|---|---|---|
| Core metaphor | OS virtual memory | Human forgetting | Benchmark | Human reading | OS segment-page | **Zettelkasten** |
| Memory structure | Flat FIFO queue | Flat storage | N/A | Episode pages | 3-tier (STM/MTM/LPM) | **Atomic notes with links** |
| Organization | None (flat) | None | N/A | Gist pages | Topical segments | **Emergent from links** |
| Key mechanism | Self-directed paging | Forgetting curve | Various tested | Gist + look-up | Heat-based eviction | **Link gen + memory evolution** |
| Retrieval | Cosine similarity | Cosine similarity (FAISS) | Cosine similarity | LLM over gists | Two-stage (segment→page) | **Cosine + linked retrieval** |
| LoCoMo Avg F1 (GPT-4o-mini) | 29.13 | 6.84 | 32.97 | 9.51 | 36.23 | **35.94** |
| Token efficiency | 16,977 | 432 | 16,910 | 643 | 3,874 | **2,520** |
| Unique strength | LLM self-manages memory | Time-based forgetting | Comprehensive benchmark | Global gist context | Persona modeling | **Memory evolution** |
| Key weakness | Topic mixing in long convos | Decay alone insufficient | N/A | No persistent memory | Complex configuration | Multi-step LLM calls per memory |

**A-Mem vs MemoryOS (the two top performers):**
- A-Mem achieves comparable F1 with **35% fewer tokens** (2,520 vs 3,874)
- MemoryOS has richer persona modeling (90-dimension user traits)
- A-Mem's memory evolution creates emergent organization; MemoryOS uses predefined segment-page structure
- A-Mem tested on 9 foundation models (including small ones); MemoryOS tested only on GPT-4o-mini

**A-Mem's key differentiator**: It's the only system where **existing memories actively change** when new memories arrive. All other systems have static storage — once something is stored, it stays the same (or gets summarized/evicted). A-Mem's memories are living, evolving entities.
