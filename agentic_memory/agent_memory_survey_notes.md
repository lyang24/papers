## Why This Survey Exists

Agent memory research has exploded, but the field is **conceptually fragmented**: different papers use "agent memory" to mean very different things (episodic, semantic, parametric, short-term, long-term, etc.), and existing taxonomies based on the simple long-term vs. short-term dichotomy cannot capture the diversity of modern systems.

This survey proposes a new **three-dimensional taxonomy** — examining agent memory through the unified lenses of **Forms**, **Functions**, and **Dynamics** — to reconcile these fragmented definitions and provide a coherent map of the landscape.

---

## Key Distinctions: Agent Memory vs. Related Concepts

### Agent Memory vs. LLM Memory

- Most of what was historically called "LLM memory" (few-shot prompting, self-reflection, KV compression) now falls under **agent memory** — these are agentic mechanisms that accumulate, transform, and reuse information.
- True **LLM-internal memory** (attention-sparsity mechanisms, recurrent-state persistence, KV-store expansions) focuses on expanding the model's representational capacity without agentic behavior, and is *outside* the scope of agent memory.

### Agent Memory vs. RAG

- **RAG** retrieves from *static, externally maintained* knowledge sources for individual inference tasks.
- **Agent memory** is instantiated within an agent's *ongoing interaction with an environment*, continuously incorporating new information generated by the agent's own actions.
- The boundary is increasingly blurred: agentic RAG (PlanRAG, Self-RAG) occupies the closest conceptual overlap — both involve autonomous interaction with information stores, but agent memory maintains an *internal, persistent, self-evolving* memory base across tasks.

### Agent Memory vs. Context Engineering

- **Context engineering** is a *resource management* paradigm — it optimizes what goes into the context window for efficient inference.
- **Agent memory** is a *cognitive modeling* paradigm — it governs what the agent *knows*, what it *has experienced*, and how these elements evolve.
- They overlap heavily in working memory (rolling summaries, token pruning), but diverge for long-lived agents where memory must persist *beyond* any single context window.

## Forms: What Carries Memory? (Section 3)

The survey identifies three fundamental representational forms:

### 1. Token-level Memory

Memory stored as explicit, discrete, human-readable units that can be individually accessed, modified, and reorganized. This is the **most common form** in existing work.

Organized by topological complexity:

| Topology | Description | Examples |
|---|---|---|
| **Flat (1D)** | Linear sequences or bags of units — no explicit inter-unit topology | MemGPT (dialogue), Expel (experience), Reflexion (trajectory), Think-in-Memory |
| **Planar (2D)** | Single-layer structured organization — graphs, trees, tables with explicit relationships but no cross-layer links | A-MEM (card-based graph), KGT (user knowledge graph), MemTree (hierarchical conversation tree) |
| **Hierarchical (3D)** | Multi-level structures with inter-layer connections — pyramids, multi-layer graphs | GraphRAG (community hierarchy), HiAgent (subgoal pyramid), HippoRAG (associative + passage layers), G-Memory (3-tier graph) |

**Trade-offs:** Flat memory is simple and scalable but lacks relational structure. Planar memory enables structured reasoning but has no vertical abstraction. Hierarchical memory is the richest but most expensive to build and maintain.

### 2. Parametric Memory

Memory embedded directly in model parameters, accessed implicitly during forward computation.

| Type | Description | Examples |
|---|---|---|
| **Internal** | Encoded in the original model weights (via pre-training, mid-training, or post-training) | Character-LM, SELF-PARAM, ROME, MEND, AlphaEdit |
| **External** | Stored in additional parameter sets (adapters, LoRA, proxy models) attached to a frozen base model | MLP-Memory, K-Adapter, WISE, ELDER, MemLoRA |

**Strengths:** Implicit, abstract, generalizable — good for role-playing, behavioral patterns, and tasks needing broad pattern induction. Zero retrieval latency.

**Weaknesses:** Updating requires retraining (costly), prone to catastrophic forgetting. Hard to audit or edit.

### 3. Latent Memory

Memory carried in internal representations (KV caches, hidden states, latent embeddings) — not human-readable but machine-native.

| Type | Description | Examples |
|---|---|---|
| **Generate** | Auxiliary models synthesize new latent representations | Gist (gist tokens), AutoCompressor, MemoRAG, MemoryLLM, MemGen, Titans |
| **Reuse** | Directly carry over prior KV caches or hidden states | Memorizing Transformers, SirLLM, FOT, LONGMEM |
| **Transform** | Compress/restructure existing latent states (selection, aggregation, projection) | Scissorhands, SnapKV, PyramidKV, H2O, RazorAttention |

**Strengths:** Privacy-preserving (opaque), token-efficient, convenient for multimodal fusion, supports rich semantic encoding.

**Weaknesses:** Black box — hard to debug, edit, or verify. States can drift or accumulate errors.

### When to Use What (Section 3.4)

| Memory Form | Best For |
|---|---|
| **Token-level** | Multi-turn chatbots, personalization, recommendation, high-stakes domains needing auditability |
| **Parametric** | Role-playing, reasoning-intensive tasks, tasks requiring fundamentally new capabilities |
| **Latent** | Multimodal agents, on-device/edge deployment, privacy-sensitive applications |

## Functions: Why Agents Need Memory? (Section 4)

The survey moves beyond the coarse long-term/short-term dichotomy and proposes three functional pillars:

### 1. Factual Memory — "What does the agent know?"

The agent's declarative knowledge base, ensuring **consistency**, **coherence**, and **adaptability** across interactions.

- **User factual memory** — facts about the user (identity, preferences, commitments)
  - *Dialogue coherence:* Preserving conversational context and persona (MemGPT, MemoryBank, RMM)
  - *Goal consistency:* Maintaining explicit task state to prevent intent drift (RecurrentGPT, A-Mem, MemGuide)
- **Environment factual memory** — facts about the external world (documents, codebases, tool capabilities)
  - *Knowledge persistence:* Structured external fact layers (HippoRAG, MemTree, LMLM, Zep)
  - *Shared access:* Common factual foundation for multi-agent collaboration (MetaGPT, G-Memory, Generative Agents)

### 2. Experiential Memory — "How does the agent improve?"

Procedural and strategic knowledge accumulated to enable **continual learning** and **self-evolution** across episodes. Classified by abstraction level:

| Type | What It Stores | Examples |
|---|---|---|
| **Case-based** | Minimally processed records — raw trajectories and solutions for direct replay | Expel, Synapse, MapCoder, Memento, JARVIS-1 |
| **Strategy-based** | Distilled insights, workflows, and reasoning patterns for cross-task transfer | Reflexion (insights), AWM (workflows), Buffer of Thoughts (patterns), ReasoningBank |
| **Skill-based** | Executable procedures — code snippets, functions, APIs, MCPs | Voyager (code library), Gorilla (API), SkillWeaver (functions), Alita/Alita-G (MCP) |
| **Hybrid** | Combines multiple abstraction levels with dynamic selection | ExpeL + strategies, G-Memory, Dynamic Cheatsheet |

### 3. Working Memory — "What is the agent thinking about now?"

Capacity-limited, dynamically controlled scratchpad for active within-episode context management.

- **Single-turn** — Processing massive immediate inputs within one forward pass
  - *Input condensation:* Hard (LLMLingua), soft (Gist, ICAE), hybrid (HyCo2) compression
  - *Observation abstraction:* Converting raw observations to structured states (Synapse for DOM, VideoAgent for video)
- **Multi-turn** — Maintaining task state and historical relevance across sequential interactions
  - *State consolidation:* Mapping growing trajectories into fixed-size state (MEM1, MemAgent, ReSum)
  - *Hierarchical folding:* Decomposing by subgoals, keeping active detail + compressed summaries (HiAgent, Context-Folding, AgentFold)
  - *Cognitive planning:* Working memory as an externalized plan or world model (PRIME, SayPlan, KARMA)

## Dynamics: How Memory Operates and Evolves (Section 5)

The memory lifecycle consists of three interconnected processes:

### 1. Memory Formation — "How to extract the memory?"

Transforming raw experience into information-dense knowledge units:

| Method | Description |
|---|---|
| **Semantic summarization** | Compressing raw data into compact summaries — incremental (MemGPT, Mem0, Mem1 with RL) or partitioned (MemoryBank by session, ReadAgent by cluster) |
| **Knowledge distillation** | Extracting specific facts (TiM, RMM) or experiential insights (ExpeL, H2R two-tier reflection, Mem-alpha learnable extraction) |
| **Structured construction** | Building topological representations — entity-level (KGT, Mem0^g, GraphRAG) or chunk-level (HAT, RAPTOR, A-MEM, CAM) |
| **Latent representation** | Encoding into machine-native vectors/KV states (MemoryLLM, M+, MemGen, CoMEM) |
| **Parametric internalization** | Fusing into model weights — knowledge editing (ROME, MEND, MEMIT) or capability training (SFT, DPO, GRPO) |

### 2. Memory Evolution — "How to refine the memory?"

Integrating new memories with the existing repository:

- **Consolidation** — Merging and abstracting
  - *Local:* Merging highly similar entries (RMM, VLN)
  - *Cluster-level:* Aligning and fusing memory clusters (PREMem, EverMemOS, TiM, CAM)
  - *Global:* Deriving system-level summaries from full experience (MOOM, Matrix, AgentFold, Context Folding)
- **Updating** — Resolving conflicts
  - *External:* Revising vector DB / KG entries (MemGPT simple replace -> Zep temporal annotations -> LightMem dual-phase -> Mem-alpha RL-based)
  - *Model editing:* Direct parameter modification (ROME causal tracing, MEMIT batch editing, MEMORYLLM self-updating)
- **Forgetting** — Removing outdated/redundant information
  - *Time-based:* Evicting oldest entries (MemGPT FIFO, exponential decay)
  - *Frequency-based:* LFU/LRU policies (XMem, KARMA Bloom filters, MemOS)
  - *Importance-driven:* Semantic valuation (Livia emotional salience, TiM/MemTool LLM-based importance scoring)

### 3. Memory Retrieval — "How to utilize the memory?"

A four-stage pipeline:

1. **Timing & Intent** — When and where to retrieve: always-on (ExpeL) vs. agent-controlled (MemGPT, MemTool) vs. fast-slow thinking (ComoRAG, PRIME) vs. latent triggers (MemGen)
2. **Query Construction** — Bridging user intent and memory index: decomposition (Visconde, PRIME, MA-RAG) or rewriting (HyDE hypothetical documents, MemoRAG draft-as-query, MemGuide intent phrases)
3. **Retrieval Strategies** — Executing the search:
   - *Lexical:* TF-IDF, BM25 — precise but brittle
   - *Semantic:* Dense embedding similarity — flexible but noisy (MemGPT, MemoryBank, Voyager)
   - *Graph:* Structure-aware traversal — multi-hop reasoning (AriGraph, EMG-RAG, HippoRAG PageRank, D-SMART beam search)
   - *Hybrid:* Combining multiple paradigms (Agent KB lexical+semantic, Semantic Anchoring, MAICC multi-factor scoring)
4. **Post-Retrieval Processing** — Refining retrieved results:
   - *Re-ranking & filtering:* Removing low-relevance items (Semantic Anchoring heuristic, learn-to-memorize RL, Memory-R1 answer agents, Memento Q-learning, MemGuide fine-tuned re-ranker)
   - *Aggregation & compression:* Merging fragments into coherent context (ComoRAG integration agent, MA-RAG extractor agent, G-Memory role-specific personalization)

## Positions and Frontiers (Section 7)

### 1. Memory Retrieval vs. Memory Generation

The field is shifting from treating memory as a static repository to be queried, toward **actively synthesizing** new memory representations on demand:
- *Retrieve then generate:* ComoRAG, G-Memory, CoMEM — retrieve first, then create a refined, context-specific representation
- *Direct generation:* MemGen, VisMem — construct latent memory tokens directly from interaction history, bypassing explicit retrieval entirely

Future generative memory should be: **context adaptive**, able to **integrate heterogeneous signals**, and **learned/self-optimizing** (via RL rather than manual rules).

### 2. Automated Memory Management

Moving from hand-crafted rules (Mem0 thresholds, A-MEM expert rules) toward agents that **autonomously manage their own memory** through explicit tool calls (add/update/delete/retrieve) and **self-optimizing memory structures** that dynamically link, index, and reorganize over time.

### 3. Reinforcement Learning Meets Agent Memory

A clear progression:
1. **RL-free** — Heuristic/prompt-driven (MemGPT, MemOS, Mem0, ExpeL, G-Memory)
2. **RL-assisted** — RL applied to selected components: re-ranking (RMM), memory writing (Mem-alpha), working memory compression (Context Folding, MemSearcher, IterResearch)
3. **Fully RL-driven** (future) — Memory architectures designed by RL, with agents having complete control over all stages of the memory lifecycle

### 4. Multimodal Memory

Most current systems are text-only. Real-world agents need memory that spans vision, audio, and other modalities. No existing system provides truly **omnimodal** memory support.

### 5. Shared Memory in Multi-Agent Systems

Evolving from isolated local memories -> centralized shared stores (MetaGPT, G-Memory) -> future **agent-aware shared memory** with role-based access control and learning-driven management policies.

### 6. Memory for World Models

Memory is the cornerstone of world models — it ensures long-term consistency in scene layout, object attributes, and motion logic. Current approaches: state-space models, explicit memory banks, and sparse memory/retrieval.

### 7. Trustworthy Memory

Three pillars: **privacy preservation** (granular permissions, differential privacy, federated access), **explainability** (traceable access paths, causal graphs of memory influence), and **hallucination robustness** (abstention under low confidence, multi-agent cross-checking).

### 8. Human-Cognitive Connections

Agent memory architectures have converged with human cognitive models — the context window + external DB mirrors Atkinson-Shiffrin's working/long-term memory split, and the factual/experiential/skill partition aligns with Tulving's episodic/semantic/procedural classification. Key divergence: human memory is *constructive* (actively reconstructing from current state), while agent memory is mostly *reproductive* (verbatim retrieval). Future systems should incorporate offline consolidation (analogous to sleep) and generative reconstruction.

## Key Takeaways

1. **Memory is a first-class primitive for agents**, not an optional add-on. It underpins long-horizon reasoning, continual adaptation, and effective interaction with complex environments.

2. **The Forms-Functions-Dynamics framework** provides a more nuanced taxonomy than traditional long-term/short-term dichotomies:
   - *Forms:* Token-level (explicit, editable) vs. Parametric (implicit, in weights) vs. Latent (implicit, in activations)
   - *Functions:* Factual (what it knows) vs. Experiential (how it improves) vs. Working (what it's thinking now)
   - *Dynamics:* Formation (extract) -> Evolution (refine) -> Retrieval (utilize)

3. **The biggest open challenges** are: fully RL-driven memory systems, truly multimodal memory, trustworthy memory with privacy/explainability guarantees, and the shift from retrieval-centric to generation-centric memory paradigms.

4. **Agent memory is distinct from** LLM memory (architectural optimizations), RAG (static knowledge retrieval), and context engineering (resource management) — though all four share substantial technical overlap and increasingly blur together.

5. **The field is moving toward** self-organizing, learnable memory systems that coevolve with the agent through reinforcement learning, rather than relying on hand-crafted rules and heuristic pipelines.
